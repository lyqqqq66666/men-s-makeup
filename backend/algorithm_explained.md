# 图像压缩算法与原理详解

本文档解释了 `image_compressor.py` 中实现的图像压缩逻辑及其背后的数字图像处理原理。

## 1. 代码实现解析

我们的压缩器 `ImageCompressor` 主要包含以下两个核心功能：

### 1.1 EXIF 方向自动纠正 (`fix_orientation`)
**问题背景**：手机或相机拍摄的照片通常包含 EXIF 元数据。其中 `Orientation` (标签 ID 274) 记录了拍摄时的方向（如横屏、竖屏）。标准的图像查看器会自动读取该标签并旋转显示，但底层图像处理库（如 Pillow）读取的是原始像素矩阵，如果不处理 EXIF，图片在处理后可能会“倒下”或旋转 90 度。

**代码逻辑**：
1.  读取图片的 EXIF 数据。
2.  检查 `Orientation` 标签的值：
    *   `3`: 旋转 180°
    *   `6`: 旋转 270° (顺时针 90°)
    *   `8`: 旋转 90° (逆时针 90°)
3.  使用 `img.rotate(angle, expand=True)` 物理旋转像素矩阵，`expand=True` 确保画布尺寸随旋转调整。

### 1.2 压缩逻辑 (`compress`)
该函数根据文件扩展名（JPG vs PNG）采用完全不同的压缩策略。

## 2. 图像压缩原理

### 2.1 JPEG 压缩原理 (有损压缩)

JPEG 是一种基于感知的有损压缩标准，利用人眼对亮度敏感而对色度不敏感的特性。

**核心步骤**：
1.  **色彩空间转换**: 将 RGB 转换为 YCbCr。Y 是亮度，Cb/Cr 是色度。算法通常会对 Cb/Cr 进行下采样（例如 4:2:0），直接减少一部分数据。
2.  **分块与 DCT (离散余弦变换)**: 将图像切分为 8x8 的像素块，进行 DCT 变换，将空间域（像素值）转换为频率域。
    *   *原理*: 图像的能量通常集中在低频部分（左上角），高频部分（右下角，代表细节和噪点）数值很小。
3.  **量化 (Quantization)** (关键步骤): 用一个预定义的量化表除 DCT 系数并取整。
    *   *代码对应*: `quality=85` 参数直接决定了量化表的精细程度。`quality` 越低，除数越大，高频系数变为 0 的越多，文件越小，但会有“马赛克”伪影。
4.  **熵编码 (Entropy Coding)**: 使用 Huffman 编码或算术编码对量化后的系数进行无损压缩。
    *   *代码对应*: `optimize=True` 告诉 Pillow 在保存时花更多时间计算最优的 Huffman 表，而不是使用标准表。这可以在不降低画质的情况下减少文件大小。

### 2.2 PNG 压缩原理

PNG 是一种支持透明通道的无损压缩格式，基于 Deflate 算法（LZ77 + Huffman）。

#### 策略 A: 无损优化 (默认)
**原理**：仅通过优化 Deflate 算法的参数（如滑动窗口大小、匹配查找深度）来减小体积，不丢弃任何像素信息。
*   *代码对应*: `optimize=True`, `compress_level=9` (最大压缩等级)。

#### 策略 B: 色彩量化 (Color Quantization, 有损)
**原理**：将 24 位真彩色（R,G,B 各 8 位，共 1600 万色）转换为 8 位索引色（Palette，仅 256 色）。
1.  **聚类**: 找出图像中最常用的 256 种颜色建立调色板。
2.  **映射**: 将每个像素的值替换为调色板中最近似颜色的索引（0-255）。
3.  **抖动 (Dithering)**: 在颜色不足的区域使用不同颜色的像素交错排列，利用人眼的“视觉混合”产生中间色的错觉，减少色带现象。

**效果**：文件大小通常减少 70% 以上，但颜色数量在极端渐变下可能不够丰富。
*   *代码对应*: `img.quantize(colors=256, method=2)`。

## 3. 为什么能保持分辨率？

我们的算法严格控制**只改变像素的编码方式**（JPEG 的量化精度或 PNG 的颜色深度），而不执行重采样（Resampling）或缩放（Scaling）操作。

*   **几何尺寸**: `Width x Height` 保持不变。
*   **信息密度**: 每个像素携带的信息量（比特数）减少了。

这就好比将一篇文章从“精装铜版纸印刷”改为“普通报纸印刷”，文字内容（分辨率）完全一样，但书的厚度和重量（文件大小）大大减轻了。
